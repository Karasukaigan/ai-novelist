遇到的问题：

electron.js 使用了 CommonJS语法（require），但我们在 package.json 中添加了 "type": "module"，这会导致冲突2.需要将 electron.js转换为 ES Module语法，或者移除 "type": "module"
preload.js 使用了 CommonJS 的 require 语法，但 package.json 中设置了 "type": "module"，导致模块系统不一致。需要将 preload.js 改为 ES6 模块语法（后续又改为cjs）
现在需要将 preload.js 重命名为 preload.cjs，因为 Electron 的 preload 脚本需要使用 CommonJS 格式，然后更新 electron.js 中的引用。
问题原因是模块系统不一致：

问题分析：

package.json 设置了 "type": "module"，使 .js 文件默认使用 ES6 模块
electron.js 使用 ES6 import 语法
但 preload.js 原来使用 CommonJS require 语法，导致 Electron 无法正确加载
修复内容：

将 preload.js 重命名为 preload.cjs（.cjs 扩展名明确表示使用 CommonJS）
恢复 preload.cjs 为 CommonJS 格式
更新 electron.js 中的 preload 路径引用

虽然 package.json 中设置了 "type": "module"，但 Electron 主进程默认不支持 ESM 模块，即使设置了 "type": "module"。

electron.js 使用了 ES6 import 语法，但 Electron 主进程需要 CommonJS 或 .mjs 扩展名才能识别 ESM。

解决方案：将 electron.js 重命名为 electron.mjs 并更新配置

在 ESM 模块（.mjs 文件）中，__dirname 和 __filename 不存在，需要手动创建。

解决方案：在 electron.mjs 顶部添加 ESM 兼容的 __dirname 定义


__dirname 和 __filename

### 6.4 最佳实践

1. **Electron 主进程**：
   - 使用 CommonJS (`.cjs`)
   - 直接使用 `__dirname` 和 `__filename`

2. **前端代码**：
   - 使用 ES Module (`.js`)
   - 使用相对路径和 `import`
   - 不需要 `__dirname` 和 `__filename`

3. **路径构建**：
   - Node.js: `path.join(__dirname, 'file.txt')`
   - 浏览器: `import file from './file.txt'`


# __dirname 和 __filename 详解

## 一、它们是什么？

### 1.1 定义

**`__dirname`** 和 **`__filename`** 是 **Node.js 的全局变量**，只在 Node.js 环境中存在。

```javascript
// __dirname: 当前文件所在的目录路径
console.log(__dirname);
// 输出示例: e:\ai-novelist\frontend

// __filename: 当前文件的完整路径（包含文件名）
console.log(__filename);
// 输出示例: e:\ai-novelist\frontend\electron.cjs
```

### 1.2 它们的作用

主要用于**构建文件路径**，特别是在 Electron 主进程中需要：

```javascript
// 加载预加载脚本
preload: path.join(__dirname, 'preload.cjs')

// 加载前端构建后的 HTML
mainWindow.loadFile(path.join(__dirname, 'dist', 'index.html'))

// 获取后端 Python 文件路径
path: path.join(__dirname, '..', 'main.py')

// 加载图标
icon: path.join(__dirname, 'assets', 'icon.png')
```

preload.cjs 文件逻辑
这个文件是 Electron 的安全桥接层，负责：

使用 contextBridge - 安全地向渲染进程（React 前端）暴露 API
使用 ipcRenderer - 处理与主进程的 IPC 通信
暴露的 API：
getBackendStatus() - 获取 Python 后端运行状态
restartBackend() - 重启后端服务
platform - 获取操作系统平台（win32/darwin/linux）
isPackaged - 判断是否为打包后的应用
项目架构
electron.cjs (主进程) ←→ preload.cjs (安全桥接) ←→ React 前端 (渲染进程)
        ↓
    Python 后端

主进程负责启动 Python 后端、创建窗口；preload 脚本作为安全层，只暴露必要的 API 给前端；前端通过 window.electronAPI 调用这些功能。


ipcRenderer.invoke = 前端发送请求给主进程
ipcMain.handle = 主进程接收并处理请求
'restart-backend' = 自定义的消息名称，两边必须一致
主进程操纵后端 = 通过 Node.js 的 spawn() 和 kill() 控制子进程

__dirname 是 Node.js 中的一个全局变量，代表当前执行文件所在的目录路径

spawn 是 Node.js 中用于创建子进程的一个重要方法，属于 child_process 模块

英文意为：产卵。编程中表示"创建/生成子进程"
基础用法：spawn(command, [args], [options])

spawn('python', ['script.py', '--port', '3000']);
[options] (可选)
spawn('npm', ['install'], {
  cwd: '/path/to/project'  // 在指定目录执行命令
});
spawn('node', ['server.js'], {
  env: {
    ...process.env,           // 继承当前环境变量
    NODE_ENV: 'production',    // 添加/覆盖变量
    PORT: '3000',
    DB_URL: 'mongodb://localhost:27017'
  }
});
// 配置标准输入输出
spawn('python', ['app.py'], {
  stdio: [
    'pipe',  // stdin: 可通过 child.stdin 写入
    'pipe',  // stdout: 可通过 child.stdout 读取
    'pipe'   // stderr: 可通过 child.stderr 读取
  ]
});
// 让子进程独立于父进程运行
const child = spawn('node', ['long-task.js'], {
  detached: true,  // 父进程退出后子进程继续运行
  stdio: 'ignore'  // 忽略标准输入输出
});
child.unref();  // 断开与父进程的引用关系
// Windows 上使用 cmd，Unix 上使用 /bin/sh
spawn('echo %PATH%', {
  shell: true  // 允许使用 shell 特性
});


path.dirname() 是 Node.js path 模块的方法，用于获取路径的目录部分（去掉文件名）。

shell: true 的作用是让 Node.js 通过系统 shell（Windows 上是 cmd.exe 或 PowerShell）来执行命令，而不是直接执行可执行文件。

两种执行方式的区别
不使用 shell (shell: false)
spawn('python', ['main.py'])

Node.js 直接在 PATH 环境变量中查找 python.exe
如果 PATH 配置有问题，可能找不到 Python
使用 shell (shell: true)
spawn('python', ['main.py'], { shell: true })

等价于在命令行执行：python main.py
Shell 会自动处理：
PATH 环境变量
命令别名（如 Windows 上 python 可能指向 py 启动器）
路径扩展

fs.existsSync(backend.path) - Node.js 的文件系统方法，检查指定路径的文件或目录是否存在
unref() 是 Node.js 中用于控制进程退出行为的核心方法。它通常与子进程（Child Process）一起使用。
具体作用

当你在 Node.js 主进程中创建了一个子进程（backendProcess），调用 unref() 后：

    允许主进程独立退出：即使这个子进程还在运行，Node.js 主进程也可以正常退出

    不会阻塞事件循环：子进程的存在不会阻止 Node.js 判断是否可以退出

相当于解绑了主进程和子进程？

webPreferences 配置是 Electron 应用的安全配置选项，用于保护渲染进程（前端页面）与主进程（Node.js 环境）之间的交互。

各配置项说明
1. nodeIntegration: false
作用：禁用渲染进程中的 Node.js 集成
效果：前端页面无法直接使用 require()、process、Buffer 等 Node.js API
原因：防止恶意代码直接访问系统资源，提高安全性
2. contextIsolation: true
作用：启用上下文隔离
效果：前端页面的 JavaScript 运行在独立的上下文中，与预加载脚本和 Electron 内部 API 隔离
原因：防止前端代码篡改 Electron 内部 API，避免原型链污染攻击
3. preload: path.join(__dirname, 'preload.cjs')
作用：指定预加载脚本路径
效果：在页面加载前执行 preload.cjs，该脚本可以安全地暴露特定的 API 给前端使用
原因：作为前端与 Electron 主进程之间的安全桥梁，只暴露必要的功能

  mainWindow.on('closed', () => {
    mainWindow = null;
  });

功能说明：

监听窗口关闭事件：mainWindow.on('closed', ...) 注册了一个事件监听器，当主窗口被关闭时触发。

释放窗口引用：mainWindow = null 将 mainWindow 变量设置为 null。

为什么需要这样做：

在 Electron 中，当 BrowserWindow 被关闭后，JavaScript 中的引用仍然存在
将变量设置为 null 可以释放对已关闭窗口的引用，避免内存泄漏
这样做也允许在需要时重新创建窗口（因为变量为 null，可以判断窗口不存在）
